# Feature Metrics Framework

기능 개발 시 **어떤 목표로, 어떤 지표를 보고, 어떻게 판단할지** 사전에 정의한다.
감이 아닌 데이터 기반으로 기능의 성공/실패를 판단하기 위한 구조.

## 문서 구조 (기능별)

각 기능에 대해 아래 구조로 작성한다:

| 섹션 | 설명 |
|------|------|
| **배경** | 왜 이 기능이 필요한가. GA/설문 등 근거 데이터 |
| **가설** | 이 기능이 어떤 행동 변화를 만들 것인가 |
| **기준선 (Baseline)** | 현재 상태의 수치 (기능 출시 전) |
| **목표 지표 (North Star)** | 이 기능의 성공을 대표하는 단일 지표 |
| **보조 지표** | North Star를 뒷받침하거나, 부작용을 감지하는 지표 |
| **추적 이벤트** | 필요한 GA4 이벤트 (기존 + 신규) |
| **성공 기준** | 출시 N주 후 이 수치면 성공/실패로 판단 |
| **모니터링** | 언제, 어떻게, 얼마나 자주 확인할지 |

## 판단 원칙

1. **기능 출시 전에 기준선을 반드시 기록한다** — 비교 없이 지표는 의미 없음
2. **North Star는 하나만 둔다** — 여러 지표가 개선되면 좋지만, 핵심 판단은 하나
3. **부작용 지표를 반드시 포함한다** — A가 올랐는데 B가 떨어지면 실패일 수 있음
4. **최소 2주 관찰한다** — 단기 변동(novelty effect)에 속지 않기 위함
5. **평일 데이터 기준으로 본다** — 주말은 DAU가 58% 하락하므로 노이즈가 큼
