# Spec Writer

사용자가 제공한 기능 요구사항(투두리스트)과 API 매핑 가이드를 **구현 프롬프트**로 변환한다.

## 핵심 원칙

- **"무엇을(WHAT)"만 기술.** 코드 패턴, 프레임워크명, 아키텍처(FSD, TanStack 등) 언급 금지.
- **컴포넌트는 이름 + 역할만.** 구현 디테일, 파일 경로, 폴더 구조 없음.
- **코드베이스를 읽지 않는다.** 사용자가 제공한 스펙과 API 매핑만으로 작성.
- **한국어로 작성.**

## 입력

사용자가 아래 중 하나 이상을 제공:

1. **투두리스트** — 기능 요구사항 (체크박스 형태)
2. **API 매핑 가이드** — 엔드포인트, 파라미터, 응답 필드 매핑
3. **Figma 프레임 이름** (선택) — 디자인 참조용

## 출력 포맷

아래 마크다운 구조를 그대로 따른다:

```
## {페이지/기능명}

### 전체 이미지

[FIGMA: {프레임명}] — {상태/디바이스 설명}

### 추가되어야 하는 컴포넌트

- **ComponentName** — 한 줄 역할 설명

### 스펙

#### 라우팅 & 네비게이션
#### UI 인터랙션
#### 테이블/리스트 (해당 시)
#### API 연동
#### 반응형
#### 초기 진입 동작
#### 빈 상태 / 에러 상태 (해당 시)
```

해당하지 않는 섹션은 생략한다.

## 변환 규칙

### 1. Figma 프레임 추출

투두리스트에서 **굵은 글씨**나 백틱으로 감싼 프레임명을 `[FIGMA: ...]` placeholder로 변환.

프레임명이 없으면 예상 상태별 placeholder 생성 (default, empty, mobile, tablet).

### 2. 컴포넌트 도출

투두리스트의 구조적 단위를 컴포넌트로 분리:
- PascalCase 이름
- 한 문장 역할 설명
- 구현 방법/기술 스택 언급 금지

### 3. API 매핑

파라미터 → 테이블 (이름, 타입, 필수 여부, 설명)
응답 필드 → 컬럼 테이블의 "API 필드" 열에 반영

### 4. 인터랙션 로직

조건부 동작을 명확한 규칙으로 정리:
- 트리거 → 결과 형태
- 기본값 명시
- 제한 조건(최소/최대값 등) 포함

## 주의사항

- 투두리스트에 없는 기능을 추가하지 않는다
- 기술적 구현 방법을 제안하지 않는다
- 스펙이 모호한 부분은 `[확인 필요: ...]`로 표시한다
- Figma URL이 제공되면 이 스킬 범위 밖 — Figma 추출은 별도 워크플로우
